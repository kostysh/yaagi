## Рамка дискавери

- Дата дискавери: `2026-03-18`
- Дата архитектурного аудита: `2026-03-23`
- Источник истины для дискавери: `docs/architecture/system.md`
- Эвристика дискавери: кандидаты сгруппированы по швам поставки и фазам реализации, а не по каждой внутренней таблице или классу; после аудита `2026-03-23` backlog дополнительно закрывает platform/toolchain/deployment seams и ownership gaps, которые архитектура считает обязательными с самого старта.
- Workflow-статусы backlog сохранены в виде protocol tokens: `candidate`, `confirmed`, `intaken`, `discarded`.
- Coverage semantics for architecture realignment: только real dossier со статусом `done` означает delivered seam; `intaken` / `confirmed` / `candidate` в backlog означают owner exists, but delivery is still deferred to that backlog line.
- Working-system invariant for this backlog: stubs, emulators, simulators, fake providers and any other placeholder runtime surfaces do not count as delivered architecture capabilities; a seam remains open until the canonical runtime path uses the real function it claims to own.
- Backlog realignment note after the delivered `F-0002` / `F-0014` review: registry/health ownership for richer organs is now delivered, but replacement of the phase-0 `vllm-fast` stub with a real `vLLM` serving path still had no explicit owner; this refresh adds `CF-023` so the backlog once again closes onto the architecture's real local-model runtime.
- Рекомендуемый backbone для архитектурно корректного старта: `CF-020 -> CF-001 -> CF-002 -> CF-003 -> CF-004 -> CF-006 -> CF-017 -> CF-007`, затем `CF-005 -> CF-008 -> CF-016 -> CF-018 -> CF-015` как завершение minimal living / first-working phase.
- Обязательный маршрут от delivered stub-backed backbone к реальному local-model runtime: `CF-010 -> CF-023 -> CF-011 -> CF-019`; без `CF-023` backlog остаётся на registry/health level и не доводит систему до настоящего containerized `vLLM` serving.
- Для внешне безопасной и операционно замкнутой working system после minimal living phase требуются ещё два явных owner seam: `CF-024` для operator auth/authz + RBAC и `CF-025` для deploy/release/rollback automation; без них delivered runtime остаётся либо локально-доверенным, либо вручную-эксплуатируемым контуром.
- Для полноценной operational/mature closure backlog также должен явно владеть support/operability contract и late phase-6 governance completion; это закреплено кандидатами `CF-026` и `CF-027`, чтобы roadmap не обрывался после technical delivery без owner-а на эксплуатацию и зрелую policy surface.

- `CF-005`: intake оформлен как [F-0011](../../ssot/features/F-0011-narrative-and-memetic-reasoning-loop.md); один dossier владеет `memetics + narrative + field journal`, while durable promotion/compaction remains with `CF-018`.

- `CF-005`: shaping must preserve the intake-fixed ownership rule that `F-0011` writes only narrative/memetic surfaces and never gains direct write access to `psm_json`, `goals`, `beliefs` or other subject-state tables.
- `CF-008`: shaping must keep `homeostat_snapshots` and bounded guardrail requests separate from `CF-015` reporting/read-model ownership; richer reports remain read-only consumers, not alternate control writers.

- `CF-009`: intake must keep `GET /state`, `GET /timeline`, `GET /episodes` and `GET /models` as bounded read-only operator surfaces over canonical owners rather than raw table dumps or alternate business-state writers.
